3/ En résumé

  • A l'intérieur de la BDD les informations sont classées, structurées et regroupées par sujet.
  • Le language de requete SQL nous permet d'échanger les informations avec la base : il est utilisé pour la communication entre le client et le serveur.
  • Dans une BDD les données sont représentées sous forme de "table"
  • Dans une table nous avons des colonnes, chaque donnée sera introduite sous forme de ligne dans la table
  • Une fois que les informations ont été enregistrées, il est important de les gérer (ajout, modification, suppression, consultation).

2/ Modélisation

Avant de creer une BDD, il est essentiel de se poser et de réléfhir à la modélisation de la base : pour éviter de se tromper.

Changer la modélisation d'une BDD en cours de production n'est pas recommandé ; cela nous forcera à réécrire des requete

Comment modéliser une BDD ?

On peut modéliser une BDD sur une feuille de papier ou on peut utiliser un logiciel pour nous aider.
Exemple : MySQL Workbench

Pour une bonne conception de BDD, il faut réfléchir en terme de sujets en modélisant ceux du monde réel. La modélisation se compose de différentes tables (une table = un sujet) ;
exemple : Si nous vendons des porduits il faut une table des produits et une table des commandes.

3/ Les tables

  • 1 sujet représente 1 table dans une base de données.
  • 1 table est un emplacement de sauvegarde.
  • Les données sont stockées à l'intérieur de tables. Une table peut être comparée à une liste, qui contient des enregistrements relatifs à un sujet bien défini.

Il faut réfléchir au sujet et à ses conséquences. Si nous vendons des produits, nous aurons certainement des membres. 1 table membre sera donc nécessaire. En terme de fonctionnalité, si nous souhaitons proposer un abonnement newsletter aux membres, nous aurons besoin d'1 table newsletter repertoriant les membres abonnés
table = 1 sujet

4/ Les colonnes et les champs

Chaque table possède généralement plusieurs champs (aussi appelés colonnes). Les colonnes / champs représentent des caractéristiques relatives au sujet (la table).

Pour savoir, quelle colonne mettre dans quelle table, il faut se poser la question suivante : qu'est-ce qui pourrait décrire mon sujet ? quelles sont les informations sur mon sujet ?
Par exemple, la table membre aura les champs : pseudo, mot de passe, nom, prenom, adresse, email, etc.

5/ Les types de données

Type de champ Description Contexte
VARCHAR Chaîne (jusqu'à 255 caractères) Pour enregistrer un pseudo, un email, le titre d'un produit ou d'un article de blog
TEXT Chaîne de caractère (illimité) Pour un contenu textuel long, un article de blog
INT Numérique Pour un identifiant, un prix, un code postal, le nombre de produits en stock etc.
DATE Date Pour la date d'un enregistrement (une inscription), la date d'une commande etc.
ENUM Une liste (une énumération) Pour proposer une liste de termes ex: "Madame","Monsieur".

6- Types de données PK AI pour les ID

a) PK (primary key = clef primaire) - Les identifiants

  • Les identifiants sont des champs particulier car ils ne décrivent pas le sujet, ce sont des champs systématiquement présents dans chaque table et ce en 1ère position.
  • Les identifiants sont uniques.
  • Chaque table possède un champs id (identifiant), surnommé PK (primary key = clef primaire).
  • Exemple : "id_membre" ou "idMembre" ou "id_user"

b) AI (Auto Increment = Auto incrémenté)

  • Auto incrémenté est une option permettant de générer un numéro unique dans un champ de type clef primaire (identifiant).
  • Par consequent, retenez bien que le 1er champ de chaque table sera systématiquement un "id" qui sera PK et AI.

c) NULL, NOT NULL

  • Dans chaque champ, nous pouvons indiquer si nous acceptons les valeurs NULL ou NOT NULL
  • NULL est un type de valeur en informatique évitant de laisser un champ vide si nous n'avons pas d'information à y mettre.

d) Les relations

Il arrive parfois que les sujets intéragissent entre eux

  • Un membre commande 1 produit, nous devons enregistreer l'information dans une table commande.
  • Un conducteur conduit 1 vehicule, il nous faut donc une table permettant de préciser qui conduit quel véhicule

Les cardinalités

  • Les cardinalités permettent de connaitre le chiffre minimum et maximum d'enregistrement pour une remation
  • Exemple :
    un homme est le fils d'une femme et d'une seule (minimum = 1 et max = 1)
    En revanche une femme peut n'avoir aucun enfant ou plusieurs (minimum = 0 et maximum = n).
  • Exemple 2 :
    1 livre possède forcément 1 auteur et qu'1 seul. Relation (1,1). Minimum 1 auteur, Maximum 1 auteur. 1 auteur peut avoir écrit plusieurs livre (ou aucun). Relation (0,n). Minimum 0 livre, Maximum N livres

Les table de jointure

  • Une table de jointure permet de faire le lien entre 2 tables.
  • Exemple : une société de taxi, qui possède des conducteurs et des véhicules.
  • 1 conducteur peut conduire 0 ou plusieurs véhicules.
  • 1 vehicule peut être conduit par 0 ou plusierus conducteurs.

9- Clefs étrangères ou FK

Les clefs étrangères sont des champs forcément clef primaires dans leur table d'origine et qui se retrouvent dans une table extérieur.

Un champ clef étrangère permet de faire la relation avec un autre sujet.

10. Les enrégistrements

Chaque enrégistrement représente une ligne dans la table de la BDD. Chaque nouvel enrégistrement va s'autoincrémenter. Un "iD" supprimé ne sera pas réutilisé !

11. Interroger la BDD

Les requêtes

Une requête permet de poser une question afin d'obtenir une réponse ou encore de donner un ordre.

Nous terminerons toutes nos requêtes SQL par un point-virgule (;) pour que MySQL sache que nous avons terminé d'écrire et qu'il doit exécuter notre demande.

De manière standard, les requêtes SQL seront écrites en majuscules SELECT * FROM membre;

Types de requêtes

Il existe 4 types de requêtes:

  • Sélection SELECT; requête question/réponse , nous demandons et obtenons une réponse
  • InsertionINSERT; requête d'action , qui aura un impact sur les données
  • ModificationUPDATE;requête d'action , qui aura un impact sur les données(mise à jour)
  • SuppressionDELETE;requête d'action , qui aura un impact sur les données (supprimer)
